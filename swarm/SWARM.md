# Teleport Contest Agent Swarm

Infrastructure for managing parallel porter agents that incrementally close
PRNG and screen divergences between our JS port and the recorded C
sessions. Optimised for high autonomy: every step is mechanically checked
against the recorded ground truth, so merges can be gated by score deltas
alone — no human PR review in the inner loop.

## The loop

```
            ┌────────────────────────────────────────────┐
            ▼                                            │
  pick-target ──▶ porter task spec                       │
                       │                                 │
                       ▼                                 │
            spawn N porter agents in worktrees ──────────┘
                       │
                       ▼
            verify-and-merge per worktree
                       │
                ┌──────┴───────┐
                ▼              ▼
              ACCEPT         REJECT
                │              │
            commit          discard
                │
                ▼
            baseline (record + advance state)
```

## CLI

All wrapped under `node swarm/bin/orchestrator.mjs <subcommand>`:

| Subcommand | What it does |
|---|---|
| `status` | Print latest vs. baseline scores, open tasks, last few runs |
| `baseline [--promote]` | Run `score.sh`, save bundle, diff against baseline. `--promote` overwrites the baseline with the current run |
| `manifest` | Regenerate `swarm/state/manifest.json` from the upstream C tree |
| `triage [--all|<session>]` | First-divergence analysis. Default prints human-readable; `--json` for machine output |
| `pick [--top=N] [--json]` | Choose the next session to attack (closest to passing first). Default emits a markdown porter task; `--json` emits the structured spec |
| `verify <worktree> --target-session=<name>` | Score a worktree, print ACCEPT/REJECT |
| `merge <worktree> --target-session=<name>` | Verify and, if ACCEPT, copy the worktree's edits onto main + commit |

## State

- `swarm/state/baseline.json` — the score bundle the merge gate compares against. Updated only when explicitly promoted (or auto on first run)
- `swarm/state/latest.json` — last full `score.sh` run; rolls forward on every merge
- `swarm/state/manifest.json` — C-to-JS port map, generated by `manifest`
- `swarm/state/state.json` — task ledger + run history

## Conventions for porter work

The Phase 2 maintainability score divides parity by `js/` diff size since
the Phase 1 freeze tag. Every porter is expected to keep this overhead small:

1. **One JS file per C file** with matching name (`monmove.c` → `js/monmove.js`).
2. **Function-by-function correspondence** inside each file (same names, same call order).
3. No abstracting away from C structure; readability beats cleverness.
4. Frozen files (`js/isaac64.js`, `js/terminal.js`, `js/storage.js`) are never touched.

## Merge gate

A worktree is accepted iff:

- Total screen-match count goes up across the public 44 sessions, AND
- No individual session loses screens, AND
- (For RNG-divergence tasks) the target session's RNG match count
  strictly increases

Any other outcome → REJECT. Worktree is discarded.

## Driving the loop

Two runners are provided. Both spawn porters in isolated git worktrees,
score each one against the full corpus, merge the best ACCEPT, and push.

### Multi-provider headless loop (`run-loop.mjs`)

This is the autonomous runner. It calls out to either `claude -p` or
`codex exec` (or both — best-of-N ensembling across providers).

```bash
# One iteration, one Claude porter + one Codex porter, push the winner:
node swarm/bin/run-loop.mjs

# Five iterations, 3 porters per round, no push (local only):
node swarm/bin/run-loop.mjs --iterations=5 --providers=claude,claude,codex --no-push

# Force a specific session as the target:
node swarm/bin/run-loop.mjs --target=sessions/seed0101-ranger-quiver-throw-travel-engrave.session.json
```

Per iteration:

1. `pick-target` chooses the session to attack.
2. Spawns N porters in parallel, each in its own git worktree.
3. Verifies each worktree against the full 44-session regression.
4. Picks the best ACCEPT (max screensDelta, zero per-session regression).
5. Cherry-picks the winner's changes onto `main` and commits.
6. `git push origin HEAD` (unless `--no-push`).
7. Cleans up losing worktrees.

The leaderboard at [mazesofmenace.ai](https://mazesofmenace.ai/) re-scores
on every push (within 2 hours).

### In-conversation drive (Claude Code Agent tool)

Inside an interactive Claude Code session you can drive iterations
manually by calling `Agent` with `isolation: "worktree"`. The same task
spec from `pick-target` works as the prompt body.

### Manual single-step (debugging)

```bash
node swarm/bin/orchestrator.mjs pick --json > task.json     # next task
node swarm/bin/run-porter.mjs --provider=claude --task-file=task.json
# → prints the worktree path on the last stdout line
node swarm/bin/orchestrator.mjs merge <worktree-path> \
     --target-session=<session-name>
```

## Provider notes

- **Claude:** uses `claude -p <prompt> --add-dir <worktree> --permission-mode bypassPermissions --model sonnet`. Auth must already be set (`claude` interactively at least once).
- **Codex:** uses `codex exec --dangerously-bypass-approvals-and-sandbox -C <worktree>`. Auth must already be set (`codex login`).

Both bypass-sandbox flags are intentional — the worktree IS the sandbox, and the merge gate is the safety net.
